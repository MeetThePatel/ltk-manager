#+TITLE: Skin Remap Cache Invalidation Overhaul

* Goal

Make generated skin remap caching correct without requiring manual
=SKIN_REMAP_VERSION= bumps during normal development.

The cache key should represent the generated remap output and the current League
game data, not an arbitrary app-side number. Startup/background work can improve
latency, but correctness must come from deterministic invalidation.

This is a hard cutover. Do not keep backwards compatibility for the old manual
version salt, do not add shims for old generated remap IDs, and do not preserve
the previous cache behavior as a fallback path. Old runtime cache entries may be
ignored or naturally orphaned.

* Current Behavior

Generated skin remaps are exposed to =ltk_overlay= as a synthetic enabled mod:

#+begin_src text
ltk:skin-remaps:<fingerprint>
#+end_src

The fingerprint is computed in =src-tauri/src/skin_remap.rs= from:

- generated WAD names
- generated override paths
- hashes of generated override bytes
- a manually maintained =SKIN_REMAP_VERSION= salt

The salt is useful as an emergency cache bust, but it is a footgun when it
becomes part of normal development. Most generator changes already change the
generated bytes or paths, which means the content fingerprint should change on
its own.

* Design

** Principle

Generated skin remap cache invalidation should be content-addressed:

- If generated override paths or bytes change, invalidate.
- If generated output is byte-for-byte identical, reuse.
- If League patches and source WAD data changes, generated output should change
  where relevant, and the overlay game-index fingerprint already invalidates
  game-layout-dependent metadata.
- A code change that does not affect generated output should not force rebuilds.

** Resolved Decisions

- Remove the manual salt entirely. The generated skin remap fingerprint should
  be purely generated content: WAD name, override path, and override bytes.
- Do not replace =SKIN_REMAP_VERSION= with an app version, commit hash, build
  hash, or schema version. Those all reintroduce non-content invalidation.
- Runtime prewarming should be automatic only when session-managed patching is
  enabled and enough settings/profile state exists to know what should be
  warmed. Low-risk data discovery, such as champion cache refresh, can happen
  whenever a League path is configured.
- Prewarm status should stay log-only. Do not add UI state unless prewarming
  later becomes user-actionable.

* Implementation Plan

** Phase 1: Hard cutover to content-only generated remap fingerprints

- Delete =SKIN_REMAP_VERSION=.
- Remove the version/salt bytes from =skin_remap_fingerprint=.
- Do not add a replacement schema version or fallback cache key.
- Leave old generated remap cache entries orphaned; the new fingerprint produces
  the new synthetic mod ID.

Acceptance:

- Existing tests pass.
- There is no manual number to bump for generated skin remap invalidation.
- The generated remap pseudo-mod ID depends only on generated override content.

** Phase 2: Prove content-derived invalidation with tests

Add focused tests around =skin_remap_fingerprint=:

- Same generated entries produce the same fingerprint.
- Changing a generated file path changes the fingerprint.
- Changing generated bytes changes the fingerprint.
- Changing WAD assignment changes the fingerprint.

If these tests require exposing a helper, keep it private to the test module
where possible.

Acceptance:

- Tests make it obvious that content changes are sufficient to invalidate the
  generated remap pseudo-mod.

** Phase 3: Remove stale versioning language

- Audit comments, logs, and nearby tests for language like "version bump",
  "schema version", or "manual cache bust".
- Update wording to "content fingerprint" where needed.
- Do not include app version or commit hash in the fingerprint. That would force
  unnecessary rebuilds when generated output is identical.

Acceptance:

- The generated remap fingerprint remains stable across app builds if generated
  output is identical.
- The plan and code no longer imply routine manual cache invalidation.

** Phase 4: Optional runtime prewarming

Add background startup work only after correctness is handled.

Potential prewarm targets:

- Game champion cache via =list_game_champions_inner= when a League path is
  configured.
- =ltk_overlay= game index for the configured League install.
- Current active profile overlay only when session-managed patching is enabled
  and a current profile exists.

Constraints:

- Prewarming must not block app startup.
- Prewarming must not start the patcher or alter the running game.
- Prewarming must be cancellable or naturally harmless if settings/profile state
  changes.
- Prewarming failures should be logged, not shown as user-facing errors unless
  the user explicitly starts patching.
- Prewarming must not be used as a correctness mechanism; cache correctness
  comes from content fingerprints and game fingerprints.

Acceptance:

- First champ-select actualization is faster after startup prewarm.
- Failed prewarm does not break manual patching.

#+begin_src sh

* Non-Goals

- Do not remove =ltk_overlay= runtime caching.
- Do not force rebuilds on every app version change.
- Do not keep backwards compatibility for old generated remap cache IDs.
- Do not add manual cache-bust shims.
- Do not make startup prewarming responsible for correctness.
- Do not start or claim the game process patcher during prewarm.
